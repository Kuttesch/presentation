Writing · Requirement Structure & Principles

What Requirements Must Cover

  • System behavior
  • Timing constraints
  • Hardware interaction
  • Error handling
  • Resource limitations
  • Public C API

Writing Principles

  • Clear and unambiguous
  • Measurable
  • Testable
  • Aware of hardware constraints
  • No implementation details

Writing · Functional & Hardware Requirements

Functional Requirements

  • Describe system behavior
  • Cover normal and fault conditions
  • Define measurable timing tolerances

Hardware Interface Requirements

  • GPIO direction and behavior
  • ADC resolution
  • SPI/I2C configuration
  • CAN frame format
  • Interrupt usage

Writing · Non-Functional & Verification

Non-Functional Requirements

  • Execution time limits with tolerance
  • Explicit RAM limits
  • No dynamic memory allocation
  • Defined CPU usage limits
  • Defined safe state behavior

Verification Strategy

  • Unit tests
  • Integration tests
  • Hardware tests / measurements
  • Static analysis
  • Code inspection

Git · Development Workflow

Goals

  • Stable production branch
  • Traceable and clean history
  • High test reliability
  • Controlled integration through review
  • Predictable development workflow

Core Rules

  • Never commit directly on main
  • Always use SSH
  • Always use signed commits
  • Write clear commit messages
  • Keep commits small and focused

Git · Issues & Branch Model

Feature Workflow

  • Create Feature Issue (Type: Feature)
  • Create Sub-Issues
  • Assign maintainer
  • Create corresponding branches

Branch Structure

  • main → protected production
  • feature/nameOfFeature
  • feature/nameOfFeature/nameOfSubIssue
  • fix/nameOfFix

Working on Sub-Issue

  • Branch from feature
  • Active development on issue branch
  • Direct commits allowed on issue branch
  • No merges outside PRs

Git · Pull Request & Review

Creating a Pull Request

  • From dev branch to merge branch
  • Assigned to maintainer
  • Reviewer assigned
  • Draft until review is expected

Review Preconditions

  • Feature fully implemented
  • ≥ 90% unit test coverage
  • All tests pass
  • CI passes

Review Criteria

  • Functional correctness
  • All sub-issues resolved
  • Minimum 90% coverage
  • Deterministic tests
  • Clear naming and small commits
  • Clean commit history
  • Signed commits
  • No architectural violations

Git · Merge & Post-Merge

Merge Conditions

  • Reviewer approval
  • All CI checks pass
  • 90%+ coverage verified

Merge Rules

  • Only assigned maintainer may merge
  • No self-merging
  • No merging without review
  • Merge without squash

Post Merge

  • Verify CI on target branch
  • Address hotfixes immediately
  • Delete development branch
  • Close issue and PR
